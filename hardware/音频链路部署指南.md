## LifeTrace 硬件实时语音链路部署指南

本指南面向需要**从硬件实时采集音频 → 通过手机转发 → 发送到 PC 上的 Audio Webhook Server → 实时转写与润色**的开发者，帮助你在本地完整跑通这一链路。

整体链路包含三部分：

1. **硬件端（Friend / Omi 录音豆）**：采集麦克风音频，通过蓝牙发送给手机。
2. **移动端 App（Android）**：与硬件配对，并通过 Webhook 将音频分片转发到 PC。
3. **PC 端 Audio Webhook Server（本仓库 `lifetrace/hardware/server.py`）**：接收音频分片，转发到 DashScope 实时 ASR，并做文本润色与落盘。

---

## 一、前置准备

- 一台可以上网的开发电脑（推荐 macOS / Windows，下面以 Windows 为例）
- 一台安卓手机（已安装 FreeTodo / Friend App，对应本仓库的移动端）
- 一颗已经到手的硬件设备（小录音豆，支持 USB-C）
- Python 环境（建议 3.10+）
- ngrok 账号（用于把本地 8000 端口暴露到公网）
- 一个 DashScope API Key（用于调用阿里云实时语音识别 + 文本润色）

目录结构中与本指南相关的关键位置：

- `hardware/other/update-xiao_nrf52840_ble_sense_bootloader-0.9.0_nosd.uf2`：Bootloader 固件
- `hardware/other/friend-xiao_nrf52840_ble_sense-1.0.4.uf2`：主固件
- `lifetrace/hardware/server.py`：Audio Webhook Server

---

## 二、给硬件刷 Bootloader 与固件

> 目标：让录音豆具备录音与通过手机发音频的能力。

### 2.1 刷入 Bootloader（类似 BIOS）

1. 用 **USB-C 数据线** 将硬件连接到电脑。
2. **快速双击复位键**（通常是板子上的小按键），硬件会以 **U 盘模式** 挂载到电脑上。
3. 打开资源管理器，你会看到一个新的磁盘（类似 `XIAO-SENSE` 或类似名字）。
4. 将 `hardware/other/update-xiao_nrf52840_ble_sense_bootloader-0.9.0_nosd.uf2` **拖拽到该磁盘根目录**。
   - 拖到一半会提示“磁盘错误”或挂载消失，这是 **正常现象**：设备在刷新 Bootloader 时自动重启了。
5. 再次双击复位键，确认设备能再次以 U 盘形式挂载即可。


### 2.2 刷入主固件

1. 再次通过 USB-C 连接好设备，并让其以 **U 盘模式** 挂载到电脑。
2. 将 `hardware/other/friend-xiao_nrf52840_ble_sense-1.0.4.uf2` 拖入该磁盘（这是当前 Demo 使用的主固件）。
3. 拖入过程中同样会发生“复制到一半磁盘消失”的情况，这是设备重启写入固件，**属正常现象**。
4. 刷写完成后，断开并重新上电，硬件会以正常工作模式启动。

## 三、在 Android 手机上配对硬件 & 配置音频 Webhook

> 目标：让手机成为「中继服务器」，从硬件接收音频，再转发给 PC。

### 3.1 与硬件配对

1. 在安卓手机上安装并打开移动端 App（本仓库的 `phone/` 工程打包出来的 APK）。
2. 进入 App 首页左上角的 **「连接设备」** 入口。
3. 点击 **「搜索设备」**，等待列表中出现名称类似 `friend-xxxx` 或 `omi-xxxx` 的设备。
4. 点击该设备，完成蓝牙配对与连接。连接成功后，App 中会显示设备已经在线。

### 3.2 打开开发者设置，启用音频 Webhook

1. 在 App 右上角找到 **设置图标**，进入设置页。
2. 下拉找到 **「开发者设置」** 区域。
3. 在开发者设置中继续下拉，找到 **「Webhooks」** 模块。
4. 在 Webhooks 中找到 **「音频字节」**（或类似命名），将其 **勾选启用**。
5. 在该项下方，会看到两个重要配置：
   - **Endpoint URL**：音频转发的目标地址
   - **发送频率（例如：每 5 秒）**

Endpoint URL 的格式示例：

- `https://<ngrok-id>.ngrok-free.app/audio?uid=test&sample_rate=16000`

其中 `/audio` 是我们 PC 端 Audio Webhook Server 暴露的 HTTP 接口，`uid` 和 `sample_rate` 会作为查询参数传给服务端。

配置完成后，点击右上角 **保存**，此时手机已具备「接收硬件音频 → 转发到指定 URL」的能力。

## 四、在 PC 上部署 Audio Webhook Server

> 目标：在本地 8000 端口启动一个 FastAPI 服务，接收音频并进行实时识别与润色。

### 4.1 安装依赖

在仓库根目录执行（推荐使用 `uv`，也可以使用 `pip`）：

```bash
cd E:\ffreeu\LifeTrace
uv sync --group dev
```


此外，Audio Webhook Server 需要：

- `fastapi`
- `uvicorn`
- `websocket-client`
- `openai`（用于调用 DashScope 兼容接口）

如果缺少，可以手动补装：

```bash
pip install fastapi uvicorn websocket-client openai
```

### 4.2 配置 DashScope API Key

Audio Webhook Server 默认使用 DashScope 的兼容 OpenAI 接口进行语音识别和文本润色。

如果未设置，`lifetrace/hardware/server.py` 中会使用一个占位的默认值，请务必替换为你自己的 Key。

### 4.3 启动 Audio Webhook Server

在仓库根目录执行：

```bash
cd E:\ffreeu\LifeTrace
python -m lifetrace.hardware.server
```

启动成功后，你会在终端看到类似输出：

- 本地地址：`http://127.0.0.1:8000`
- 重要接口：
  - `POST /audio`：手机通过 Webhook 转发的音频入口
  - `GET /`：健康检查
  - `GET /recordings`：最近录音列表
  - `GET /transcripts`：最近转录文本列表

所有的音频与转录文件会落盘到：

- `lifetrace/hardware/audio_recordings/pcm/`
- `lifetrace/hardware/audio_recordings/wav/`
- `lifetrace/hardware/audio_recordings/transcripts/`

## 五、使用 ngrok 将本地 8000 端口暴露到公网

> 目标：让手机可以通过公网 URL 调用你本地的 `/audio` 接口。

### 5.1 安装并登录 ngrok

1. 前往 `https://ngrok.com` 注册并下载客户端。
2. 安装完成后，在命令行中配置 Authtoken（按 ngrok 官网指引执行一次性登录命令）。

### 5.2 启动 HTTP 隧道

在本机上执行：

```bash
ngrok http 8000
```

启动成功后，你会看到类似输出：

- `https://<random-id>.ngrok-free.app -> http://localhost:8000`

记下这个 `https://<random-id>.ngrok-free.app`，后续需要填到手机 App 的 Webhook 配置中。

### 5.3 配置手机端 Endpoint URL

在安卓 App 的 **开发者设置 → Webhooks → 音频字节** 中，将 Endpoint URL 设置为：

```text
https://<random-id>.ngrok-free.app/audio?uid=test&sample_rate=16000
```

其中：

- `/audio` 与 `lifetrace/hardware/server.py` 中的路由保持一致
- `uid` 可以是真实用户 ID，也可以先用测试值
- `sample_rate` 建议与硬件端一致（通常为 `16000`）

保存后，手机会按你配置的频率（例如每 5 秒）向该 URL 发送一次音频分片。

## 六、端到端验证步骤

1. **确认服务都已启动**
   - 硬件上电运行（或带在身上）
   - 手机 App 已连接硬件，并在 Webhooks 中启用了「音频字节」
   - PC 上 Audio Webhook Server 已在本地 8000 端口运行
   - ngrok 已将 `https://<random-id>.ngrok-free.app` 映射到本地 `http://127.0.0.1:8000`
2. **开始说话**
   - 带着录音豆正常说话，对着设备讲话。
3. **在 PC 终端观察日志**
   - 你会看到类似：
     - `bytes_received`、`duration_seconds` 的日志
     - ASR 识别到的句子（`...` 中间结果，`✓` 最终结果）
     - LLM 润色后的文本（前面带有 📝）
4. **在浏览器中验证**
   - 打开 `http://127.0.0.1:8000/recordings` 查看最近录音
   - 打开 `http://127.0.0.1:8000/transcripts` 查看最近转录文本预览

如果能够看到新的录音与转录内容，说明 **硬件 → 手机 → ngrok → PC → DashScope** 的整条链路已经跑通。

## 七、常见问题与排查

### 7.1 手机端提示无法访问 Webhook

- 确认 ngrok 进程仍在运行，且隧道地址未过期。
- 在浏览器中直接访问 `https://<random-id>.ngrok-free.app/`，确认能返回 Audio Webhook Server 的根路径 JSON。

### 7.2 终端无任何请求日志

- 检查手机 App 的 Webhook 配置是否**保存成功**。
- 确认 Endpoint URL 是否以 `/audio` 结尾，并携带正确的 `uid` 与 `sample_rate` 参数。
- 确认硬件与手机连接正常，录音状态在 App 中为「进行中」。

### 7.3 DashScope 返回错误 / 文本不更新

- 检查 `DASHSCOPE_API_KEY` 是否正确，且有足够调用额度。
- 如网络不稳定，可暂时将 `TextPolisher` 中的 LLM 调用关闭，仅打印 ASR 原始结果即可。

### 7.4 延迟与切片长度如何选择？

- 当前实现示例为 **每 5 秒推送一次** 音频分片。
- 分片太短：切分过于碎片化，ASR 上下文不连贯，识别质量差。
- 分片太长：整体延迟变大，不够实时。
- 可以根据自己的业务需求，在手机端 Webhook 配置中调整推送周期，并在 `server.py` 中根据切片长度优化拼接与润色策略。
