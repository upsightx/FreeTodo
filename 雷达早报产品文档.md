# 雷达早报产品文档

## 一、市场需求

### 1.1 核心痛点

在当前信息爆炸的时代,用户面临着严重的信息过载问题:

- **信息分散**:有价值的信息分散在多个平台(微博、B站、抖音、知乎等),用户需要逐一访问各平台
- **算法绑架**:现有门户网站和信息流产品由平台算法控制,用户被动接收推荐内容
- **效率低下**:用户需要花费大量时间筛选、过滤与自己需求无关的信息
- **个性化缺失**:缺乏真正以用户需求为中心的信息聚合工具

### 1.2 目标用户

- 科研工作者:需要追踪特定领域的最新进展
- 专业人士:关注行业专家、意见领袖的观点
- 知识工作者:需要高效获取定制化信息的人群

### 1.3 产品价值主张

**"重新掌控自己的信息流"**

雷达早报提供两大核心价值:
- **信息聚合**:跨平台整合用户关心的所有信息源
- **定制化**:用户完全控制"要看什么"和"不看什么"

这是目前市场上尚未有产品能够实现的能力。

## 二、技术架构

### 2.1 整体架构

雷达早报基于MediaCrawlerPro作为技术基座,作为整个雷达项目生态中的一个独立产品模块,同时与其他模块(如"找人"功能、金融场景插件等)保持联动。

```
雷达项目基座(MediaCrawlerPro)
    ├── 早报模块(本产品)
    ├── 找人/找用户模块
    └── 具体场景应用(如金融插件)
```

### 2.2 核心技术流程

产品采用"抓取-过滤-精炼"三阶段处理流程:

#### **阶段一:智能抓取**

```
用户自然语言输入
    ↓
大语言模型解析
    ↓
提取关键词 + 识别博主ID
    ↓
多平台并行搜索
    ↓
原始信息片段集合
```

- **多维度信息源**:
  - 按平台维度:支持微博、B站、抖音、知乎等多平台
  - 按博主维度:可关注特定内容创作者(通过博主ID)
  - 按关键词维度:基于主题领域进行内容筛选

- **自然语言配置**:用户无需手动填写结构化参数,通过一段自然语言描述即可完成配置。例如:
  > "我比较关心跟迁移学习相关的一些东西,尤其关注李沐和三巨头的一些行为,但我不希望看到广告内容。"

#### **阶段二:智能过滤**

```
原始信息片段
    ↓
黑名单过滤(已读/不感兴趣主题)
    ↓
大语言模型内容评估
    ↓
展示决策(是/否)
```

- 过滤维度:
  - 去重:已读内容自动过滤
  - 黑名单:用户明确不想看的内容类型
  - 内容质量:通过LLM评估内容与用户需求的匹配度

#### **阶段三:内容精炼与呈现**

```
过滤后的信息片段
    ↓
大语言模型生成Overview摘要
    ↓
排版优化
    ↓
类报纸式早报界面
```

- **智能摘要**:为用户生成当日所关心话题的总体概览
- **优雅呈现**:采用类似报纸的版式设计,提供良好的阅读体验

### 2.3 定时任务机制

**前一天后台整点爬取 + 次日定时展示**

```
Day N (前一天):
    用户开机 → 后台爬虫程序启动
    → 利用用户电脑开机时间(可能长达10+小时)
    → 慢速、低压力地持续爬取内容（整点爬取）

Day N+1 (当天):
    用户开机 → 立即展示已处理完成的早报
    → 同时启动新一轮后台爬取
```

**设计优势**:
- 降低爬虫压力,避免短时间大量请求被平台检测
- 用户无感知等待,开机即可查看
- 充分利用前一天的空闲时间进行数据采集

### 2.4 技术栈

- **前端**:AI 秘书
- **后台爬虫**:MediaCrawlerPro爬虫引擎(支持多平台)
- **AI能力**:大语言模型(用于自然语言解析、内容过滤、摘要生成)
- **任务调度**:定时任务系统

## 三、功能与产品需求

### 3.1 核心功能模块

#### **功能1:自然语言配置界面**

**需求描述**:
- 提供单一文本输入框,支持用户用自然语言描述信息需求
- 通过Prompt工程引导大语言模型解析出:
  - 关键词列表
  - 博主ID列表
  - 黑名单内容类型

**用户交互**:
```
输入示例:
"我关心AI领域的技术进展,尤其是LeCun和Hinton的最新观点,
但不想看论文解读类的长文"

系统解析结果:
✓ 关键词: ["AI", "技术进展", "大模型"]
✓ 博主: ["LeCun", "Hinton"]  
✓ 黑名单: ["论文解读", "长文"]
```

**技术实现要点**:
- 设计专门的Prompt模板用于信息提取
- 支持多平台博主ID的识别与匹配
- 兼容简单关键词输入(如用户只输入1-2个词)

#### **功能2:后台智能爬虫系统**

**需求描述**:
- 在用户电脑后台静默运行
- 根据用户配置的关键词和博主进行多平台内容抓取
- 支持长时间慢速爬取(前一天10+小时持续运行，整点爬取)

**技术要求**:
- 爬虫需低资源占用,不影响用户正常使用
- 实现跨平台搜索API封装
- 数据本地缓存机制

#### **功能3:智能内容过滤引擎**

**需求描述**:
- 对爬取的每一条信息片段进行智能评估
- 根据用户黑名单和内容质量进行过滤决策
- 去除重复和已读内容

**过滤逻辑**:
```python
for 信息片段 in 爬取结果:
    if 信息片段 in 已读历史:
        continue
    
    if LLM评估(信息片段, 用户黑名单) == "不匹配":
        continue
    
    if LLM评估(信息片段, 用户关键词) == "匹配":
        加入展示队列
```

#### **功能4:早报生成与展示**

**需求描述**:
- 每日定时(用户打开网页后)生成个性化早报
- 包含智能生成的Overview总结
- 采用优雅的类报纸排版

**界面布局**:
```
┌─────────────────────────────────┐
│  [日期] 您的个性化早报           │
├─────────────────────────────────┤
│  📰 今日概览                     │
│  [AI生成的当日重点事件摘要]      │
├─────────────────────────────────┤
│  📌 信息流                       │
│  ├─ [信息1:标题+摘要+来源]       │
│  ├─ [信息2:标题+摘要+来源]       │
│  └─ ...                         │
└─────────────────────────────────┘
```

**技术实现要点**:
- 使用LLM生成Overview摘要(格式:"您关心的[topic]今日发生了...")
- 信息卡片支持跳转到原始链接
- 支持用户反馈(喜欢/不喜欢)以优化后续推荐

### 3.2 产品开发优先级

**Phase 1(MVP核心功能)**:
1. ✅ 自然语言输入框 + LLM关键词提取
2. ✅ 基于关键词的单平台爬虫(先支持一个主流平台)
3. ✅ 简单黑名单过滤
4. ✅ 基础早报展示页面

**Phase 2(功能增强)**:
1. 多平台支持(微博、B站、抖音、知乎)
2. 博主ID识别与追踪
3. 后台定时任务优化(前一天爬取机制)
4. LLM驱动的内容质量评估

**Phase 3(体验优化)**:
1. Overview智能摘要生成
2. 优雅的报纸式排版
3. 用户反馈机制
4. 历史早报归档

### 3.3 用户体验关键指标

- **便捷性**:用户只需一段话即可完成配置
- **准确性**:展示的信息与用户需求高度相关
- **时效性**:每日定时更新,信息不过时
- **非侵入性**:后台运行不影响用户工作

## 四、与生态系统的集成

雷达早报作为独立产品模块,同时与AI秘书生态的其他功能协同:

- **数据联动**:早报中识别的人物可与"找人"模块联动
- **场景扩展**:未来可为特定场景(如金融分析)提供定制化早报
- **用户增长**:作为AI秘书的特色功能吸引用户

## 五、商业价值

### 5.1 用户增长策略

- 产品核心价值(重新掌控信息流)是市场空白点
- 通过视频推广展示产品特色功能
- 融入AI秘书生态,提升整体用户体量

### 5.2 未来扩展方向

- **多模态内容**:支持视频内容的智能摘要
- **人机交互研究**:结合眼动仪等设备,研究用户注意力模式
- **科研价值**:基于产品积累的数据可产出人机交互、信息检索等方向的学术论文

## 附录:开发注意事项

1. **迭代式开发**:每完成一个小功能模块及时演示,避免理解偏差
2. **爬虫合规性**:采用慢速爬取策略,避免触发平台反爬机制
3. **用户体验优先**:产品设计的核心是降低用户配置成本,提升内容相关性
4. **可扩展架构**:为未来接入更多平台和功能预留接口
