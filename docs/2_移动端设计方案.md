# FreeU 移动端设计方案

> 移动端在 FreeU 架构中的定位是**辅助终端 IO**——它不承载重度计算，但在感知补全和用户触达两个维度上不可或缺。
> 本文档按**职能定义 → UI 设计 → 技术实现**的顺序组织。

---

# 第一部分：职能定义

## 一、职能总览

移动端承担两大类职能：


| 职能大类      | 核心定位           | 一句话描述                          |
| --------- | -------------- | ------------------------------ |
| **感知采集**  | 系统的眼睛和耳朵（移动场景） | 把手机能感知到的一切，以及硬件设备感知到的一切，汇入中心节点 |
| **观察与操作** | 用户的随身控制面板      | 让用户随时随地查看 AI 产出、管理待办、做出决策      |


此外，移动端还包含一个**全局设置页**，作为感知权限、隐私、账号与设备的统一配置入口（非核心交互职能，不单独归类）。

```
┌─────────────────────────────────────────────────────────────┐
│                       移动端 Mobile APP                      │
├──────────────────────────┬──────────────────────────────────┤
│    职能一：感知采集        │     职能二：观察与操作             │
│                          │                                  │
│  ① 硬件桥接转发           │  ① 信息查看                      │
│     蓝牙 ← 硬件设备       │     通知推送 / 待办列表            │
│     HTTP → 中心节点        │                                  │
│                          │  ② 轻量操作                      │
│  ② 手机原生感知           │     卡片交互 / 待办管理            │
│     截图/拍照 → AI 分析    │     快捷对话                      │
│     GPS / 传感器数据       │                                  │
│     手机麦克风             │  ③ 设置（全局）                   │
│     剪贴板 / 通知栏        │     感知权限 / 隐私 / 账号         │
├──────────────────────────┴──────────────────────────────────┤
│                          │                                  │
│                   远程安全通道（HTTPS）                        │
│                          ▼                                  │
│                  中心存算一体节点                              │
└─────────────────────────────────────────────────────────────┘
```

## 二、感知采集

> 核心原则：移动端只负责采集和转发原始/半加工数据，所有重度处理（融合、记忆写入、意图识别）交给中心节点。

### 2.1 硬件桥接转发

手机是硬件设备接入系统的**必经桥梁**——硬件通过蓝牙连接手机，手机通过 HTTPS 转发到中心节点。


| 维度        | 说明                  |
| --------- | ------------------- |
| **连接方式**  | 蓝牙 BLE（低功耗蓝牙）连接硬件设备 |
| **转发目标**  | 中心存算一体节点（PC 或独立节点）  |
| **转发协议**  | HTTPS / WSS（远程安全通道） |
| **支持的硬件** | 录音豆、智能耳机、未来 AI 眼镜等  |


**数据流：**

```
硬件设备（录音豆/耳机/眼镜）
       │  蓝牙 BLE
       ▼
    移动端 APP
       │  ① 接收音频流/传感器数据
       │  ② 可选：本地做轻量预处理（如 VAD 静音检测，减少无效传输）
       │  HTTPS / WSS（远程安全通道）
       ▼
  中心存算一体节点
       │  ASR → 感知流 → 后续全流程
       ▼
```

**关键能力：**

- **蓝牙设备管理**：扫描、配对、连接状态监控、断线重连
- **数据缓冲**：网络不稳定时本地临时缓存，恢复后自动补传
- **流量控制**：根据网络状况（WiFi/4G）动态调整传输策略
- **后台保活**：确保蓝牙连接和数据转发在 APP 后台时不被系统杀死

### 2.2 手机原生感知

手机本身就是一个强大的感知模块，拥有 PC 不具备的传感器和场景能力。

#### a) 截图/拍照 → AI 分析（用户主动触发）

用户看到任何感兴趣的内容，截图或拍照后发送给 AI 分析。


| 维度       | 说明                                          |
| -------- | ------------------------------------------- |
| **触发方式** | 用户手动截图/拍照，或从相册选取                            |
| **处理流程** | 图片 → 上传至中心节点 → 多模态 LLM 分析 → 结果回传展示          |
| **典型场景** | 拍下白板内容自动整理为笔记；截图一段对话让 AI 分析要点；拍摄名片自动录入联系人信息 |


#### b) GPS / 位置信息


| 维度       | 说明                                                                      |
| -------- | ----------------------------------------------------------------------- |
| **采集方式** | 后台定期上报（低频，如每 5-15 分钟）                                                   |
| **用途**   | 为中心节点提供用户当前位置上下文——辅助日程提醒（"你快到公司了，今天上午有个会"）、场景感知（"你在咖啡厅，要不要整理一下刚才会议的笔记"） |
| **隐私控制** | 默认关闭，用户可选择启用；精度可调（精确/模糊/仅城市级别）                                          |


#### c) 手机麦克风（可选）


| 维度         | 说明                                     |
| ---------- | -------------------------------------- |
| **触发方式**   | 用户主动开启语音输入，或唤醒词触发                      |
| **处理流程**   | 音频流 → 上传中心节点 → ASR + 后续流程              |
| **与硬件的区别** | 硬件麦克风是持续被动采集（用户无感），手机麦克风是用户主动触发或特定场景唤醒 |


#### d) 剪贴板监听（可选）


| 维度       | 说明                            |
| -------- | ----------------------------- |
| **采集方式** | 用户复制内容时，APP 检测到剪贴板变化          |
| **用途**   | 用户复制一个地址、一段文本、一个链接时，自动作为上下文补充 |
| **隐私控制** | 默认关闭，需用户显式授权                  |


#### e) 系统通知栏监听（可选，Android 优先）


| 维度       | 说明                                                              |
| -------- | --------------------------------------------------------------- |
| **采集方式** | 通过无障碍服务/通知监听服务，读取其他 APP 的推送通知                                   |
| **用途**   | 自动感知微信消息、日历提醒、邮件摘要等，不需要用户手动截图                                   |
| **平台差异** | Android 可通过 NotificationListenerService 实现；iOS 受限，仅能通过快捷指令等间接方式 |
| **隐私控制** | 需系统级权限授权，用户可选择监听哪些 APP                                          |


### 2.3 感知数据汇入感知流

移动端采集的所有数据，最终都以 `PerceptionEvent` 的形式汇入中心节点的感知流（Perception Stream）：


| 感知源          | source 枚举值            | modality | 备注                |
| ------------ | --------------------- | -------- | ----------------- |
| 硬件麦克风（经手机转发） | `mic_hardware`        | audio    | 手机只做转发            |
| 手机截图/拍照      | `mobile_image`        | image    | 上传图片 + LLM 分析后的文本 |
| GPS 位置       | `mobile_gps`          | text     | 经纬度 + 地理编码文本      |
| 手机麦克风        | `mic_mobile`          | audio    | 用户主动触发            |
| 剪贴板          | `mobile_clipboard`    | text     | 用户复制的内容           |
| 通知栏          | `mobile_notification` | text     | 其他 APP 的推送内容      |

## 三、观察与操作

> 核心原则：移动端是用户离开电脑后与 AI 保持连接的窗口。展示以"消费"为主（查看），操作以"轻量"为主（确认/编辑/回复），复杂操作引导回 PC。

### 3.1 信息查看

#### a) 推送通知接收与展示


| 维度       | 说明                                                |
| -------- | ------------------------------------------------- |
| **内容来源** | 中心节点检测到需要用户关注的事件后，主动推送到手机                         |
| **展示形态** | 系统推送通知（摘要） + APP 内富通知卡片（完整上下文）                    |
| **典型场景** | "导师刚在飞书问你论文进度" / "明天下午的会议和打球冲突了" / "你和王总已经三个月没联系" |


与 PC 端通知卡片的对比：


| 维度       | PC 端            | 移动端               |
| -------- | --------------- | ----------------- |
| **触发场景** | 用户在电脑前          | 用户离开电脑            |
| **信息密度** | 高（富交互卡片，含完整上下文） | 中等（先推送摘要，点击展开详情）  |
| **交互深度** | 深（采纳/编辑/稍后/忽略）  | 轻（快速确认/稍后/查看详情跳转） |


#### b) 待办列表


| 维度     | 说明                            |
| ------ | ----------------------------- |
| **内容** | AI 自动提取的 + 用户手动添加的待办事项        |
| **展示** | 按优先级/时间排序的列表视图，支持筛选（今日/本周/全部） |
| **状态** | 待处理 / 进行中 / 已完成 / 已忽略         |


### 3.2 轻量操作

#### a) 通知卡片交互


| 操作        | 说明                          |
| --------- | --------------------------- |
| **确认/采纳** | 接受 AI 建议，触发后续执行（如发送回复、创建日程） |
| **稍后处理**  | 暂时搁置，加入待处理队列                |
| **忽略**    | 跳过此条通知                      |
| **查看详情**  | 展开完整上下文，或跳转到 APP 内相应模块      |


#### b) 待办管理


| 操作       | 说明              |
| -------- | --------------- |
| **标记完成** | 勾选完成某个待办项       |
| **简单编辑** | 修改待办的标题、时间、优先级  |
| **手动添加** | 快速新建一个待办项       |
| **语音录入** | 说一句话，AI 自动解析为待办 |


#### c) 快捷对话


| 操作       | 说明                    |
| -------- | --------------------- |
| **快速提问** | 向 AI 提问，AI 基于全量上下文回答  |
| **语音对话** | 通过手机麦克风与 AI 语音交互      |
| **截图提问** | 截图 → 发给 AI → AI 分析并回答 |


### 3.3 设置（全局设置页）

设置页不属于核心交互职能，而是作为 APP 的全局配置入口，统一管理所有偏好与权限。


| 设置项        | 说明                          |
| ---------- | --------------------------- |
| **感知权限管理** | 开关各项感知能力（GPS、剪贴板、通知栏、麦克风等）  |
| **隐私与数据**  | 查看/删除 AI 记忆数据，数据保留策略        |
| **感知全局开关** | 一键暂停/恢复所有感知采集（PC + 硬件 + 手机） |
| **通知偏好**   | 推送频率、免打扰时段、通知分级             |
| **账号与设备**  | 登录/登出、设备绑定管理、中心节点连接配置       |

## 四、MVP 功能优先级

基于"轻量模式（PC 兼任私有云）"的 MVP 架构，移动端功能分优先级推进：

### P0 — MVP 必须（最小可用）


| 功能               | 所属职能  | 理由                      |
| ---------------- | ----- | ----------------------- |
| 硬件蓝牙桥接转发         | 感知采集  | 硬件设备接入系统的唯一通道，缺此则硬件无法工作 |
| 推送通知接收与展示        | 观察与操作 | 用户离开电脑后接收 AI 主动服务的唯一渠道  |
| 通知卡片交互（确认/稍后/忽略） | 观察与操作 | 用户需要能对 AI 的推送做出响应       |
| 待办列表查看           | 观察与操作 | 随时查看 AI 提取的待办事项         |


### P1 — 体验增强


| 功能            | 所属职能  | 理由                |
| ------------- | ----- | ----------------- |
| 截图/拍照 → AI 分析 | 感知采集  | 手机作为感知模块的核心差异化能力  |
| 快捷对话（文本+语音）   | 观察与操作 | 移动场景下与 AI 交互的自然方式 |
| 待办简单编辑与手动添加   | 观察与操作 | 让待办管理闭环           |


### P2 — 远期扩展


| 功能       | 所属职能  | 理由                                                     |
| -------- | ----- | ------------------------------------------------------ |
| GPS 位置上报 | 感知采集  | 丰富上下文，但隐私敏感，需谨慎推进                                      |
| 剪贴板监听    | 感知采集  | 补充感知维度，但各平台限制较多                                        |
| 通知栏监听    | 感知采集  | 自动化程度最高的感知方式，但权限门槛高                                    |
| 知识库搜索与浏览 | 观察与操作 | 感知数据本身就是知识库的一部分，移动端优先保证数据采集和通知触达，知识库的完整检索与浏览体验在 PC 端更优 |

# 第二部分：UI 设计

## 五、整体结构：底部 4 标签页

```
┌─────────────────────────────────────────────┐
│  ┌─────────────────────────────────────────┐│
│  │          顶部状态栏（全局常驻）            ││
│  │  🟢 已连接核心节点    🎧 录音豆已连接      ││
│  │  ⏸ 感知已暂停（仅暂停时显示）              ││
│  └─────────────────────────────────────────┘│
│                                             │

│             各标签页内容区域                   │

│                                             │
│  ┌─────────┬─────────┬─────────┬─────────┐ │
│  │  消息    │  待办    │  对话    │  我的   │ │
```

> **设计原则**：前两个 Tab（消息、待办）面向"AI 找你"的场景；第三个 Tab（对话）面向"你找 AI"的场景；第四个 Tab（我的）是低频管理。用户 80% 的时间花在前三个 Tab。

### 6.3 Tab 3：对话（AI 交互 + 手机感知入口）

| **对话记录**  | 保留历史对话，可上滑回看                      |
| **感知集成**  | 拍照/相册/截图按钮直接嵌在输入栏，用户发送图片后 AI 自动分析 |

## 七、顶部状态栏

所有标签页共享一个**全局状态栏**，实时展示系统关键状态：


| 状态项        | 展示规则                         |
| ---------- | ---------------------------- |
| **核心节点连接** | 🟢 已连接 / 🔴 未连接（未连接时突出提示）    |
| **硬件设备**   | 已连接的设备名称 + 电量（无设备时不显示）       |
| **感知状态**   | 仅在感知暂停时显示 ⏸ 提醒，正常运行时不显示（不打扰） |

# 第三部分：技术实现

## 九、后台保活

移动端最大的技术挑战是**蓝牙转发需要后台常驻**：


| 平台          | 策略                                 |
| ----------- | ---------------------------------- |
| **Android** | 前台服务 + 通知栏常驻提示，利用蓝牙 BLE 连接保活       |
| **iOS**     | 利用 Core Bluetooth 后台模式，配合后台刷新和推送唤醒 |


---

## 十、网络连接策略

移动端统一通过远程安全通道连接中心节点，不区分局域网/外网场景，简化连接逻辑：


| 维度       | 策略                                                    |
| -------- | ----------------------------------------------------- |
| **连接方式** | HTTPS / WSS 远程连接（通过内网穿透工具如 ZeroTier、Tailscale，或公有云中转） |
| **断线处理** | 本地缓存队列，网络恢复后自动补传                                      |
| **心跳检测** | 定期检测中心节点在线状态，离线时提示用户                                  |


---

## 十一、数据安全

- 所有传输走 HTTPS/WSS 加密通道
- 本地临时缓存加密存储，上传成功后自动清除
- 敏感感知数据（GPS、剪贴板）需用户显式授权

---

## 十二、跨平台方案

MVP 阶段建议使用跨平台框架（如 Flutter / React Native），一套代码同时覆盖 Android 和 iOS，降低开发成本。蓝牙和系统级能力通过平台原生插件桥接。（这里需要确认一下目前的代码的技术方案 如果不好迁移就使用原有的代码）
