# 后端架构改造方案

> 本文档描述 FreeTodo 后端的完整现状盘点，以及向四模块架构演进的改造思路。

---

## 一、当前后端现状

### 1.1 技术栈

- **框架**：FastAPI
- **数据库**：SQLite + SQLModel（SQLAlchemy ORM）
- **配置**：Dynaconf（热重载）
- **LLM**：OpenAI 兼容 API（支持阿里云 DashScope）
- **Agent 框架**：Agno（工具调用、学习记忆、AgentOS 服务化）
- **后台任务**：APScheduler
- **实时通信**：WebSocket
- **向量检索**：ChromaDB + sentence-transformers
- **OCR**：RapidOCR + WinRT OCR
- **ASR**：DashScope 语音识别

### 1.2 代码结构

```
lifetrace/
├── server.py                # 入口（FastAPI 应用 + 生命周期管理）
├── agent_os.py              # AgentOS 独立服务入口
├── core/
│   └── module_registry.py   # 模块注册系统（28 个模块，插件式加载）
├── routers/                 # API 路由层（30+ 文件）
│   └── chat/                # 对话子模块（6 种模式）
├── services/                # 业务逻辑层（18 个服务）
├── storage/                 # 数据访问层（15+ Manager）
├── llm/                     # LLM 集成层（30+ 文件）
│   ├── agno_tools/          # Agno 工具集（14 个待办工具）
│   ├── agno_plan/           # Agno 计划系统
│   └── tools/               # 通用工具注册
├── jobs/                    # 后台定时任务（15+ 文件）
│   └── proactive_ocr/       # 主动 OCR 子系统
├── perception/              # 感知流模块（已实现）
├── schemas/                 # 请求/响应数据结构
├── util/                    # 工具函数
└── migrations/              # 数据库迁移
```

### 1.3 分层架构

```
Router (API 端点)  →  Service (业务逻辑)  →  Manager (数据访问)  →  Database (SQLite)
                                            ↕
                                    LLM / Agent 层
```

## 三、改造策略：绞杀者模式

### 3.1 核心思路

**不删旧代码、不改旧代码、不维护旧代码。新功能全部用新模块实现。**

### 3.2 三类处置

```
┌─────────────────────────────────────────────────────────────────┐
│  ✅ 继续维护（2.1 ~ 2.3）                                       │
│     基础设施 + 感知层 + Agno Agent                               │
│     → 这是新架构的骨架和执行引擎                                 │
├─────────────────────────────────────────────────────────────────┤
│  🔄 保持运行（2.4）                                              │
│     前端依赖的 API（todo、chat、screenshot 等）                  │
│     → 前端能用就行，不开发新功能                                 │
├─────────────────────────────────────────────────────────────────┤
│  ⛔ 不再维护（2.5 ~ 2.7）                                       │
│     被替代的模块 + 边缘功能                                      │
│     → 自然淘汰，新模块成熟后可逐步关闭                           │
└─────────────────────────────────────────────────────────────────┘
```

### 3.3 感知流是切割点

```
┌──────────────────────────────────────────────────┐
│         旧代码（⛔ 不维护 + 🔄 保持运行）          │
│  recorder → OCR → Event → Activity → RAG         │
│  audio → ASR → Transcription → Todo提取           │
│      │              │                             │
│      └──── 数据写入感知流 ───┘                     │
└──────────────────┬───────────────────────────────┘
                   │
          ┌────────┴────────┐
          │     感知流        │  ← 切割点
          │ PerceptionEvent  │
          └────────┬────────┘
                   │
┌──────────────────┴───────────────────────────────┐
│           新代码（✅ 新建模块）                     │
│                                                   │
│  memory/ → intent/ → planner/ → Agno Agent       │
└───────────────────────────────────────────────────┘
```

## 四、新建三个模块

### 4.1 目录结构

```
lifetrace/
├── perception/            # 已有，继续维护
│
├── memory/                # 新建：记忆模块
│   ├── __init__.py
│   ├── writer.py          # 写入：订阅感知流，保存原始数据
│   ├── compressor.py      # 压缩：事件聚合 + 图结构组织
│   ├── indexer.py         # 索引：结构化 + 语义 + 图索引
│   ├── updater.py         # 更新：动态学习、记忆演化
│   ├── models.py          # 新数据模型（MemoryEvent, CausalEdge 等）
│   └── manager.py         # Memory 模块生命周期管理
│
├── intent/                # 新建：意图识别模块
│   ├── __init__.py
│   ├── screener.py        # 小模型初筛
│   ├── reasoner.py        # 大模型深度推理
│   ├── models.py          # 意图数据模型
│   └── manager.py         # 意图识别模块管理
│
├── planner/               # 新建：规划执行模块
│   ├── __init__.py
│   ├── orchestrator.py    # 编排：组装上下文 + 自然语言指令
│   ├── approval.py        # 审批：风险分级 + 通知卡片确认
│   ├── models.py          # 规划执行数据模型
│   └── manager.py         # Planner 模块管理
```

### 4.2 数据流

```
感知流 (PerceptionEvent)
    │
    ▼
memory/writer → 原始写入（新表）
memory/compressor → L0→L1→L2→L3 多级压缩 + 因果图谱
memory/indexer → 多维检索能力就绪
    │
    ├─── intent/screener 订阅感知流，实时小模型初筛
    │        │
    │        ▼ 触发
    │    intent/reasoner + memory/indexer → 大模型深度推理
    │        │
    │        ▼ 识别出意图
    │    planner/orchestrator → 从 Memory 检索上下文 → 组装自然语言指令
    │        │
    │        ▼
    │    planner/approval → 风险分级 → 通知卡片审批
    │        │
    │        ▼ 确认
    │    Agno Agent（已有）→ LLM 规划 + 工具执行 → 结果推送
    │
    └─── 用户主动对话 → Agno Agent 直接处理（已有链路，不变）
```

### 4.3 关键约束

1. **新表隔离**：新模块使用独立的数据表，不修改旧的 `models.py`
2. **感知流是唯一数据入口**：新模块通过订阅感知流获取数据，不直接读取旧模块的表
3. **模块注册**：每个新模块在 `module_registry.py` 中注册，遵循现有插件机制
4. **前端兼容**：旧 API 继续运行，新模块提供新 API 端点，前端逐步迁移

## 五、实施顺序

| 阶段 | 目标 | 内容 |
|:---|:---|:---|
| **Phase 1** | Memory 写入+压缩 | 实现 writer（订阅感知流写入新表）和 compressor（事件聚合），跑通写入→压缩链路 |
| **Phase 2** | Memory 索引 | 实现 indexer（结构化+语义+图索引），让数据可检索 |
| **Phase 3** | 意图识别 | 实现 screener + reasoner，从感知流实时检测意图，结合 Memory 做 RAG |
| **Phase 4** | 规划执行 | 实现 orchestrator + approval，编排上下文丢给 Agno Agent，加审批闭环 |

每个 Phase 独立可交付，完成后即可验证效果。
