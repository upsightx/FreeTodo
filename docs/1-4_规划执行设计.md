# 规划执行设计

> 本文档定义规划执行模块的设计方案。
> 规划和执行能力由 **Agno Agent（LLM）** 提供——用自然语言描述任务，Agent 自己规划并执行。
> 本模块的职责是**编排和管控**：组装上下文、触发 Agent、审批把关。

---

## 一、双链路模式

系统同时支持两条触发路径，共享同一个执行引擎（Agno Agent）：

```
┌─────────────────────────────────────────────────────────────────┐
│                                                                 │
│  链路 1（被动，已有）                                             │
│  用户主动对话 ──→ Agno Agent 直接规划+执行 ──→ 结果              │
│                                                                 │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  链路 2（主动，新建）                                             │
│  感知流 ──→ 意图识别 ──→ planner 编排 ──→ 审批 ──→ Agno Agent  │
│                              │                        执行      │
│                         从 Memory                               │
│                         注入上下文                               │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

### 链路 1：被动模式（用户主动对话）

用户通过前端 AI 对话直接与 Agno Agent 交互。Agent 收到自然语言指令后，自行推理、调用工具、返回结果。

```
用户："帮我创建一个明天下午开会的待办"
  → Agno Agent 推理 → 调用 create_todo → 返回结果
```

这条链路**已经实现**，不需要改动。

### 链路 2：主动模式（系统自动触发）

系统通过感知流持续监测用户环境，检测到需要关注的事件后，主动触发服务。

```
感知流检测到："导师在微信问你论文进展"
  → 意图识别：社交传递型意图，需要回复
  → planner 编排：从 Memory 检索上下文 + 组装自然语言指令
  → 审批：通知卡片展示方案，用户确认
  → Agno Agent 执行：起草回复、查找文件等
```

这条链路是**新建的核心能力**。

### 两条链路的关系

| | 链路 1（被动） | 链路 2（主动） |
|:---|:---|:---|
| **触发** | 用户说一句话 | 感知流 + 意图识别自动触发 |
| **上下文** | 用户当次对话内容 | 感知流 + Memory 长期记忆 |
| **审批** | 无，用户自己发起所以直接执行 | 有，系统主动触发需用户确认 |
| **执行引擎** | Agno Agent | Agno Agent（同一个） |
| **现状** | 已实现 | 待建 |

## 二、Planner 模块职责

Planner 不实现规划逻辑本身（规划由 LLM/Agno Agent 完成），它的职责是**编排和管控**：

### 2.1 组装上下文

将意图识别的输出 + Memory 中的相关记忆，拼装成一段高质量的自然语言指令：

```
意图识别输出：导师在微信问论文进展
    +
Memory 检索：
  · 上周你说过"本周五之前完成初稿"
  · 论文草稿在 Desktop/论文v3.docx
  · 上次和导师聊过第三章的数据问题
    │
    ▼
组装为自然语言指令：
  "导师张教授在微信问我论文进展，请根据以下背景帮我：
   1. 起草一段回复
   2. 查找论文相关文件
   背景信息：[上周承诺周五完成初稿][草稿位置][历史对话记录]"
    │
    ▼
丢给 Agno Agent 规划+执行
```

**上下文喂得越好，Agent 干得越好。** 这也是 Memory 模块必须先建的原因。

### 2.2 审批把关

Agent 生成方案后，planner 拦截结果，根据风险等级决定是否需要用户确认：

| 风险等级 | 操作类型 | 审批方式 |
|:---|:---|:---|
| **低风险** | 信息检索、生成摘要、关联联想 | 自动执行，结果通知 |
| **中风险** | 建议回复、创建日程、文件查找 | 通知卡片确认后执行（用户可采纳、编辑、稍后或忽略） |
| **高风险** | 发送消息、操作文件、调用外部服务 | 必须用户明确授权（用户可采纳、编辑、稍后或忽略） |

### 2.3 完整流程

```
意图识别输出
    │
    ▼
┌──────────────────────────────────────┐
│  Planner 编排                         │
│                                      │
│  1. 从 Memory 检索相关上下文           │
│  2. 组装自然语言指令                   │
│  3. 发送给 Agno Agent                 │
└──────────────────┬───────────────────┘
                   │
                   ▼
┌──────────────────────────────────────┐
│  Agno Agent（已有）                    │
│                                      │
│  LLM 推理：目标拆分 + 方案生成         │
│  工具调用：执行具体操作                │
└──────────────────┬───────────────────┘
                   │
                   ▼
┌──────────────────────────────────────┐
│  审批回路（四种交互模式）              │
│                                      │
│  低风险 → 自动放行，结果通知          │
│  中/高风险 → 通知卡片：               │
│    采纳 → 直接执行                    │
│    编辑 → 跳转主窗口修改后执行        │
│    稍后 → 加入待处理队列              │
│    忽略 → 跳过                        │
└──────────────────┬───────────────────┘
                   │
                   ▼
              结果推送给用户
```
