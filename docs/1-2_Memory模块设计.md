# Memory 模块设计

> 本文档定义个人记忆系统（Memory）的架构设计与技术方案选型。
> Memory 的输入来自 [感知流](./1-1_感知流设计.md) 输出的结构化感知数据（PerceptionEvent）。

---

## 一、设计目标

Memory 系统要解决一个核心问题：**把用户一天中发生的所有事情忠实记录下来，并在需要时能准确找回来。**

具体目标：

| 目标 | 说明 |
|:---|:---|
| **全量保存** | 感知流产出的所有事件不丢弃，先存后处理 |
| **高效压缩** | 原始碎片化数据逐级聚合为有意义的事件/活动/摘要，控制存储膨胀 |
| **精准检索** | 支持按人/事/时间/语义多维度找到目标记忆 |
| **动态演化** | 新事件自动关联历史，记忆随使用强化/淡化 |
| **本地优先** | 数据默认存储在本地，不上云，保障隐私 |
| **人类可读** | 存储格式用户可直接阅读和编辑 |

---

## 二、核心链路（已确认）

```
感知流（PerceptionEvent）
    │
    ▼
  写入（实时保存原始数据）
    │
    ▼
  压缩（碎片 → 事件 → 活动 → 摘要）
    │
    ▼
  索引（结构化 + 语义 + 关系检索）
    │
    ▼
  更新（自动关联、强化/淡化、用户修正）
```

无论采用哪种技术方案，这条链路是固定的。两种方案的区别在于**"数据用什么结构存、怎么组织、怎么检索"**。

---

## 三、写入层设计（两种方案共用）

Memory 的第一步是**忠实保存感知流的原始输出**，作为一切后续处理的基底。

### 写入内容

| 字段 | 说明 |
|:---|:---|
| 原始文本 | 感知流融合后的结构化文本（PerceptionEvent.content_text） |
| 元信息 | 时间戳、来源通道（屏幕/音频）、来源 APP、说话人标签 |
| 原始快照 | 可选保留原始截图/音频片段引用，用于回溯验证 |

### 写入原则

- **全量写入、先存后处理**：原始数据不做任何丢弃，压缩在后续环节异步执行
- **本地优先**：原始数据默认存储在本地
- **流式追加**：感知流持续产生，写入为 append-only 模式

---

## 四、技术方案对比

我们提出两种技术路径，分别实现并对比效果：

| 维度 | 方案 A：Flat File（Memory-as-File） | 方案 B：Graph Structure（图结构存储） |
|:---|:---|:---|
| **核心思路** | 记忆 = 文件，一天一个 Markdown，事件即文本行 | 记忆 = 图，事件是节点，关系是边 |
| **存储介质** | 本地 Markdown 文件（.md） | 图数据库（轻量级，如 SQLite + 邻接表，或 NetworkX） |
| **人类可读性** | 极高，用户可直接用任何编辑器查看/编辑 | 中等，需要工具辅助查看，原始存储非直接可读 |
| **实现复杂度** | 低，文件读写即可 | 中高，需维护节点/边/索引 |
| **压缩方式** | LLM 读取文件内容，生成更高级别的摘要文件 | LLM 分析节点，合并/生成更高层级节点 |
| **检索方式** | 文本搜索 + Embedding 向量检索 | 结构化查询 + 图遍历 + 向量检索 |
| **关系追溯** | 弱，需要 LLM 推理文本间关系 | 强，因果链/关联关系显式存储在边上 |
| **可扩展性** | 文件数量增长后检索变慢 | 图索引天然支持大规模关联查询 |

## 五、方案 A：Flat File（Memory-as-File）

### 5.1 核心理念

**Memory = 一组 Markdown 文件**。这是最朴素的方案——人的记忆就像日记本，一天一页，发生什么就记什么。检索时让 LLM 直接读文件内容。

灵感来源：[Basic Memory](https://github.com/basicmachines-co/basic-memory)、Obsidian 笔记系统、Letta 的 Memory-as-File 思路。

### 5.2 存储结构

```
memory/
├── raw/                          # L0 原始流（一天一个文件）
│   ├── 2026-02-24.md
│   ├── 2026-02-23.md
│   └── ...
├── events/                       # L1 事件（压缩后生成）
│   ├── 2026-02-24.md
│   └── ...
├── activities/                   # L2 活动（更高级别聚合）
│   ├── 2026-W09.md               # 按周聚合
│   └── ...
├── summaries/                    # L3 摘要
│   ├── 2026-02-monthly.md
│   └── ...
├── entities/                     # 实体档案（按人/项目等维度）
│   ├── 张教授.md
│   ├── 论文项目.md
│   └── ...
└── index/                        # 向量索引文件（Embedding 缓存）
    └── faiss.index
```

### 5.3 L0 原始流文件格式

每个 PerceptionEvent 写入为一行（或一个块），一天一个文件（具体格式根据实际感知流输出调整，此处仅作示意）：

```markdown
# 2026-02-24 感知记录

## 09:15 | mic_pc | self
嗯今天先把论文第三章的实验数据整理一下

## 09:17 | ocr_screen | wechat | 张教授
论文进展如何了？上周说的实验数据补充了吗？

## 09:18 | mic_pc | self
导师问了，我得赶紧回他

## 09:20 | ocr_screen | wechat | self → 张教授
张老师好，实验数据正在整理中，预计今天下午能完成初步分析

## 10:05 | ocr_screen | vscode
打开文件：experiment_analysis.py，正在编写数据处理代码

## 10:32 | ocr_screen | chrome
搜索：matplotlib 双Y轴绑定不同坐标范围
```

**格式约定：**
- 每条记录格式：`## HH:MM | source | [speaker/app] [→ target]`
- 正文紧跟在标题下方
- 一天结束时文件自动封存，新一天创建新文件

### 5.4 压缩流程

```
                  L0 原始流（一天一个 .md）
                         │
            ┌────────────┤  异步/定时触发
            ▼            │
    LLM 读取当天原始文件   │
    提取事件、归纳摘要     │
            │            │
            ▼            │
    L1 事件文件（一天一个） │
    每个事件一个段落       │
            │            │
            ▼            │
    L2 活动文件（一周一个） │
    多个相关事件归纳为活动  │
            │            │
            ▼            │
    L3 摘要文件（月度/季度）
```

**L1 事件文件示例：**

```markdown
# 2026-02-24 事件摘要

## Event: 与导师微信沟通论文进展
- **时间**: 09:17 - 09:25
- **参与人**: 张教授, 我
- **来源**: 微信（屏幕感知 + 麦克风）
- **摘要**: 导师询问论文进展，我回复实验数据正在整理中，预计下午完成
- **待办**: 下午完成实验数据初步分析
- **关联**: #论文项目 #张教授

## Event: 编写实验数据分析代码
- **时间**: 10:05 - 11:30
- **来源**: VSCode（屏幕感知）
- **摘要**: 在 VSCode 中编写 experiment_analysis.py，实现数据处理和可视化
- **关联**: #论文项目 #编程
```

### 5.5 检索策略

| 检索方式 | 实现 | 适用场景 |
|:---|:---|:---|
| **时间检索** | 直接定位日期对应的文件 | "昨天发生了什么" |
| **关键词搜索** | 全文搜索 raw/ 和 events/ 目录 | "找到关于张教授的对话" |
| **语义检索** | 对 L1 事件做 Embedding，存入向量索引，相似度召回 | "之前有人跟我说过一个数据可视化工具" |
| **实体检索** | 读取 entities/ 下对应人物/项目的档案文件 | "我和张教授的所有互动历史" |

**检索流程：**

```
用户查询 / 系统自动召回
        │
        ├─ 解析时间范围 → 定位对应日期文件
        ├─ 提取关键词 → 全文搜索
        ├─ 语义 Embedding → 向量相似度 Top-K
        │
        ▼
    候选记忆片段（多路召回合并去重）
        │
        ▼
    LLM 精排 + 生成回答
```

### 5.6 优势与风险

**优势：**
- 实现极简，文件读写即核心逻辑
- 人类可读可编辑，用户可直接打开文件查看/修改自己的记忆
- 天然兼容各种文本工具（grep、编辑器、Obsidian 等）
- LLM 直接读 Markdown 效果好，无需格式转换
- 存储格式不依赖任何数据库，迁移方便

**风险：**
- 关系追溯能力弱：事件之间的因果链、关联关系全靠 LLM 现场推理，无法显式遍历
- 检索效率随数据增长下降：半年以上的数据量可能导致搜索变慢
- 压缩质量依赖 LLM：摘要的准确性和一致性取决于 LLM 能力
- 去重困难：同一事件在不同层级文件中可能存在冗余

## 六、方案 B：Graph Structure（图结构存储）

### 6.1 核心理念

**Memory = 因果记忆图谱**。事件是节点，事件之间的关系（因果、关联、从属）是边。图结构天然支持"这件事是因为什么导致的"这类追溯查询。

灵感来源：Agentic Memory、MemoryScope (Alibaba)、知识图谱。

### 6.2 存储结构

采用 SQLite 作为底层存储（本地轻量，无需额外服务），上层用图语义组织：

```
memory/
├── memory.db                     # SQLite 数据库（节点表 + 边表 + 元信息表）
├── raw/                          # L0 原始流仍以文件形式保存（作为不可变基底）
│   ├── 2026-02-24.md
│   └── ...
├── vectors/                      # 向量索引
│   └── faiss.index
└── exports/                      # 可选：导出为人类可读格式
    └── ...
```

### 6.3 数据模型

**节点（Node）表：**

```sql
CREATE TABLE nodes (
    node_id     TEXT PRIMARY KEY,     -- UUID
    node_type   TEXT NOT NULL,        -- 'raw' | 'event' | 'activity' | 'summary' | 'entity'
    level       INTEGER NOT NULL,     -- 0=原始流, 1=事件, 2=活动, 3=摘要
    title       TEXT,                 -- 节点标题
    content     TEXT NOT NULL,        -- 节点内容（结构化文本）
    timestamp   TEXT NOT NULL,        -- 事件发生时间
    time_end    TEXT,                 -- 事件结束时间（可选）
    source      TEXT,                 -- 来源通道
    app         TEXT,                 -- 来源 APP
    persons     TEXT,                 -- 涉及人物（JSON 数组）
    tags        TEXT,                 -- 标签（JSON 数组）
    importance  REAL DEFAULT 0.5,     -- 重要度（0-1），随访问/引用动态变化
    created_at  TEXT NOT NULL,
    updated_at  TEXT NOT NULL
);
```

**边（Edge）表：**

```sql
CREATE TABLE edges (
    edge_id     TEXT PRIMARY KEY,
    from_node   TEXT NOT NULL REFERENCES nodes(node_id),
    to_node     TEXT NOT NULL REFERENCES nodes(node_id),
    relation    TEXT NOT NULL,        -- 'caused_by' | 'followed_by' | 'related_to' | 'part_of' | 'about'
    confidence  REAL DEFAULT 1.0,     -- 关系置信度（0-1）
    reason      TEXT,                 -- 为什么认为有这个关系
    created_at  TEXT NOT NULL
);
```

**索引：**

```sql
CREATE INDEX idx_nodes_type ON nodes(node_type);
CREATE INDEX idx_nodes_time ON nodes(timestamp);
CREATE INDEX idx_nodes_persons ON nodes(persons);
CREATE INDEX idx_edges_from ON edges(from_node);
CREATE INDEX idx_edges_to ON edges(to_node);
CREATE INDEX idx_edges_relation ON edges(relation);
```

### 6.4 节点层级

| 层级 | node_type | 粒度 | 说明 | 示例 |
|:---|:---|:---|:---|:---|
| **L0** | raw | 单条感知记录 | 感知流原始输出，全量保留 | "14:03 屏幕-微信：导师发来'论文进展如何'" |
| **L1** | event | 一个完整交互/活动 | 多条 L0 聚合为一个事件 | "14:00-14:15 与导师微信对话，讨论论文进展" |
| **L2** | activity | 一段时间内的主题聚合 | 多个 L1 归纳为一个活动 | "下午：论文修改工作（含导师沟通+文献查阅+写作）" |
| **L3** | summary | 日/周级别 | 更高层级的浓缩 | "本周重点推进论文第三章，卡在实验数据分析" |
| **—** | entity | 实体档案 | 人物/项目/概念的长期画像 | "张教授：我的导师，研究方向是…" |

### 6.5 图结构示例

```
[张教授] ──(about)──▶ [导师布置论文任务]
                              │
                      (caused_by)
                              │
                              ▼
                     [论文第三章写作] ◄──(part_of)── [论文项目 Activity]
                              │
                      (followed_by)
                              │
                              ▼
                     [与导师讨论进展]
                              │
                      (caused_by)
                              │
                              ▼
                     [补充实验数据]
```

### 6.6 压缩流程

```
L0 原始感知事件写入 nodes 表（node_type='raw'）
        │
        │  定时/异步触发
        ▼
    LLM 分析近期 L0 节点
        │
        ├─ 聚合为 L1 Event 节点
        │   └─ 添加 part_of 边连接到 L0
        │
        ├─ 推断因果关系
        │   └─ 添加 caused_by / followed_by 边
        │
        ├─ 关联已有 Entity 节点
        │   └─ 添加 about 边
        │
        ▼
    周期性触发更高级别压缩
        │
        ├─ L1 → L2 Activity（多事件归纳）
        └─ L2 → L3 Summary（日/周摘要）
```

### 6.7 检索策略

| 检索方式 | 实现 | 适用场景 |
|:---|:---|:---|
| **结构化查询** | SQL 查询 nodes 表（按时间/人物/标签过滤） | "上周三和张教授的对话" |
| **语义检索** | 对 L1 节点做 Embedding，向量相似度召回 | "之前有人提过一个可视化工具" |
| **图遍历** | 沿边遍历（BFS/DFS），支持链式追溯 | "这个任务最初怎么来的"（沿 caused_by 回溯） |
| **混合检索** | 结构化 + 语义 + 图遍历的多路召回 | 复杂问题 |

**图遍历检索示例：**

```
查询："补充实验数据 这件事最初是怎么来的？"

[补充实验数据]
    │ caused_by
    ▼
[与导师讨论进展]
    │ caused_by
    ▼
[论文第三章写作]
    │ caused_by
    ▼
[导师布置论文任务]  ← 根因

回答：这件事最初源于导师布置的论文任务 → 你开始写第三章
     → 和导师讨论时他建议补充实验数据 → 产生了这个待办
```

### 6.8 优势与风险

**优势：**
- 关系显式存储，因果追溯和关联查询是原生能力
- 结构化索引效率高，SQL 查询性能稳定，数据增长不影响检索速度
- 节点层级天然支持多粒度压缩
- Entity 节点可构建长期人物/项目画像
- 可以做复杂的图分析（如：最近哪些事件形成了闭环、哪些任务链断了）

**风险：**
- 实现复杂度高于文件方案
- 原始数据在数据库中，用户不能直接用文本编辑器查看（需要额外的导出/可视化工具）
- 关系推断依赖 LLM 质量，错误的边会污染图结构
- SQLite 的图查询能力有限（多跳遍历需要递归 CTE 或应用层实现）

## 七、动态更新（两种方案共用逻辑）

无论采用哪种方案，Memory 都不是静态存档，而是持续演化的系统。

### 7.1 自动关联

新事件写入时，系统主动"回顾"历史，发现新旧事件之间的联系：

- **方案 A**：LLM 读取最近的事件文件 + 相关实体档案，在文件中以 `#tag` 和 `关联: [...]` 标注关系
- **方案 B**：LLM 分析后直接在图中添加边，关系机器可读

### 7.2 记忆演化

| 演化行为 | 说明 | 方案 A 实现 | 方案 B 实现 |
|:---|:---|:---|:---|
| **合并** | 高度相关的事件合并为更高层级 | 生成新的 Activity/Summary 文件 | 创建新层级节点 + part_of 边 |
| **强化** | 被反复检索/引用的记忆提升权重 | 在文件头部标注访问计数 | 更新 importance 字段 |
| **淡化** | 长期未访问的低重要度记忆降低优先级 | 检索时按日期衰减排序 | importance 随时间衰减 |
| **修正** | 新信息推翻旧结论 | 编辑对应文件内容 | 更新节点内容 / 修改边的置信度 |

### 7.3 用户可干预

| 能力 | 方案 A | 方案 B |
|:---|:---|:---|
| 查看记忆 | 直接打开 .md 文件 | 通过界面/命令行导出 |
| 编辑记忆 | 直接编辑文件 | 通过界面/API 修改 |
| 删除记忆 | 删除文件行或整个文件 | 删除节点（级联删除相关边） |
| 标注关系 | 手动添加 `关联: [...]` 标签 | 通过图谱可视化界面添加边 |

## 八、MVP 实现计划

### 8.1 MVP 定义：方案 A（Memory-as-File）

**MVP 阶段只实现方案 A**，即纯 Markdown 文件持久化方案。不引入图结构，不引入数据库。

**决策依据：**

| 考量 | 说明 |
|:---|:---|
| **最快可用** | 文件读写即核心逻辑，无需图数据库建模、边关系维护等复杂实现 |
| **人类可读** | 用户直接用编辑器打开 .md 文件查看和修改自己的记忆，这是产品差异化 |
| **LLM 友好** | 大模型直接读 Markdown 效果好，无需格式转换，天然支持 RAG |
| **零依赖** | 不依赖 SQLite 图查询、NetworkX 等额外组件，降低部署复杂度 |
| **可演进** | 文件方案积累的数据是通用格式，后期如需引入图结构可平滑迁移 |

### 8.2 MVP 功能范围

```
MVP（方案 A）：
├── L0 写入：感知流 → 每日一个 .md 文件，append-only
├── L1 压缩：一天结束时 LLM 自动生成事件摘要文件
├── 基础检索：时间定位 + 关键词全文搜索
├── Agent Skill：封装为 MemoryToolkit，供对话 Agent 调用
└── 实体档案：可选，entities/ 下按人/项目维度归档

不做（留给后续阶段）：
├── × 图结构存储（方案 B）
├── × 向量语义检索（Embedding + FAISS）
├── × L2/L3 多级压缩
├── × 记忆强化/淡化机制
└── × 自动关联发现
```

### 8.3 阶段演进路线

```
Phase 1 — MVP（当前）：方案 A 基础版
├── L0 原始流写入（一天一个 .md）
├── L1 事件压缩（异步/定时）
├── 基础检索（时间 + 关键词）
└── MemoryToolkit（Agent Skill）

Phase 2 — 检索增强
├── 向量语义检索（Embedding + FAISS/ChromaDB）
├── Entity 实体档案自动维护
├── 多路召回 + LLM 精排
└── 接入主动服务 Agent 的 RAG

Phase 3 — 可选引入图结构（按需决定）
├── 评估方案 A 在检索和关系追溯上的瓶颈
├── 如需图能力：L1+ 以上引入图结构（混合方案）
├── 因果链/关联关系显式存储
└── 图遍历检索

Phase 4 — 动态演化
├── 自动关联（新旧事件联系发现）
├── 记忆强化/淡化机制
├── 用户干预界面
└── L2/L3 多级压缩
```

### 8.4 混合方案的可能性（远期参考）

两种方案并不互斥。一种务实的思路是：

- **L0 原始流用 Flat File 存储**（方案 A）：保持人类可读，写入最简
- **L1+ 以上用图结构组织**（方案 B）：事件节点和关系存入数据库，支撑检索和追溯
- 两层之间通过节点 ID 与原始文件行号的映射关联

这样既保留了原始数据的可读性，又获得了结构化检索的能力。是否引入混合方案，在 Phase 2 检索增强阶段根据实际瓶颈决定。
