# 分布式部署设计：Center-Sensor 架构

> 本文档描述 LifeTrace 的多机分布式部署方案，将感知采集与核心计算彻底分离。

---

## 一、背景与动机

当前 LifeTrace 采用单机模式（`0_整体架构实现.md` 中的"轻量部署模式"），一台 PC 同时承担感知采集、核心计算、数据存储和用户交互。当用户拥有多台设备时，希望：

1. 一台常开的**中心节点**（台式机或云端）承担所有重度计算和数据存储
2. 日常使用的 **PC** 只做屏幕/音频感知采集，用户通过浏览器远程访问中心节点的 UI
3. 未来硬件端（录音豆、耳机、AI 眼镜）也能接入同一套体系

**核心目标**：PC 变为最薄的客户端——只运行一个感知守护进程 + 打开浏览器。

---

## 二、架构总览

### 2.1 统一接入模型

Center 对外只暴露**两个感知入口**，所有终端设备统一接入：

```
                                         ┌──────────────────────────────────────────────────────┐
                                         │              Center (中心存算节点)                      │
                                         │                                                      │
┌────────────┐                           │  ┌──────────────────────┐  ┌───────────────────────┐ │
│ PC 浏览器   │  音频 WS ══════════════▶  │  │  音频入口 (WS)        │  │  文本入口 (HTTP)       │ │
│ (麦克风)   │                           │  │  /ws/audio/transcribe │  │  /api/perception/ingest│ │
└────────────┘                           │  │                      │  │                       │ │
                                         │  │  接收 PCM 音频流      │  │  接收 PerceptionEvent  │ │
┌────────────┐                           │  │  → ASR 转文字        │  │  (已转为文本的感知数据) │ │
│ PC Sensor  │  HTTP POST ─────────────▶ │  │  → PerceptionStream  │  │  → PerceptionStream    │ │
│ (截屏OCR)  │                           │  └──────────┬───────────┘  └───────────┬─────────────┘ │
└────────────┘                           │             │                          │               │
                                         │             └────────────┬─────────────┘               │
┌────────────┐                           │                          ▼                             │
│ 录音豆      │  音频 WS ══════════════▶  │                   PerceptionStream                     │
│ (麦克风)   │                           │                          │                             │
└────────────┘                           │               ┌──────────┼──────────┐                  │
                                         │               ▼          ▼          ▼                  │
┌────────────┐                           │           Memory     Intent     Planner → Agent       │
│ AI 眼镜    │  音频 WS ══════════════▶  │                                                      │
│ (未来)     │                           └──────────────────────────────────────────────────────┘
└────────────┘
```

**设计原则**：

- **音频类**数据（麦克风、系统音频）→ 统一走 `/ws/audio/transcribe` WebSocket
- **文本类**数据（OCR、主动 OCR）→ 统一走 `/api/perception/ingest` HTTP POST
- 不同设备只是 `source` 字段不同，Center 端处理逻辑完全统一
- 硬件端、PC 端、未来移动端共用同一套接口，无需为每类设备单独开发

### 2.4 角色定义

| 角色 | 定位 | 运行的组件 |
|:---|:---|:---|
| **standalone** | 当前单机模式（默认） | 全部模块 + 本地前端 |
| **center** | 中心存算节点 | 全部模块 + 前端 + 感知事件接收端点 |
| **sensor** | 感知守护进程 | 截屏/OCR/主动OCR + 事件转发（无 FastAPI） |

## 三、统一感知接入协议

### 3.1 两条通道

| 通道 | 协议 | 端点 | 数据格式 | 适用场景 |
|:---|:---|:---|:---|:---|
| **音频通道** | WebSocket | `/ws/audio/transcribe` | PCM 16kHz 单声道二进制帧 | 所有音频来源 |
| **文本通道** | HTTP POST | `/api/perception/ingest` | PerceptionEvent JSON | 所有已转文本的感知数据 |

### 3.2 音频通道（WebSocket）

**已有接口**，无需新建——所有音频设备复用同一个 `/ws/audio/transcribe`。

```
建连时携带 source 参数：
  /ws/audio/transcribe?source=mic_pc&node_id=pc-home
  /ws/audio/transcribe?source=mic_hardware&node_id=bean-01

数据帧：
  → Binary: PCM 16kHz 16bit LE 音频帧
  ← JSON:   { "header": {"name": "TranscriptionResultChanged"},
              "payload": {"result": "...", "is_final": true} }

控制帧：
  → JSON:   { "type": "stop" }
  → JSON:   { "type": "init", "is_24x7": true, "source": "mic_hardware" }
```

不同音频来源的区别仅在 `source` 参数：

| source | 来源 | 接入方 |
|:---|:---|:---|
| `mic_pc` | PC 电脑麦克风 | PC 浏览器（现有） |
| `mic_hardware` | 硬件麦克风（录音豆/耳机） | 硬件设备 / 移动端转发 |
| `system_audio` | PC 系统音频 | Sensor 守护进程（未来） |

### 3.3 文本通道（HTTP POST）

**新增接口**——接收已经转为文本的感知数据。

```
POST /api/perception/ingest
Content-Type: application/json

Body: PerceptionEvent JSON
{
  "source": "ocr_screen",
  "modality": "text",
  "content_text": "微信聊天记录：张三说...",
  "content_raw": null,
  "metadata": {
    "node_id": "pc-home",
    "app": "wechat",
    "window_title": "微信"
  },
  "priority": 0
}

Response: { "ok": true, "event_id": "uuid", "sequence_id": 42 }
```

批量接口：

```
POST /api/perception/ingest/batch
Body: {
  "node_id": "pc-home",
  "events": [ PerceptionEvent, ... ]
}
Response: { "ok": true, "count": 5 }
```

### 3.4 设备接入矩阵

| 设备 | 音频通道 (WS) | 文本通道 (POST) | 说明 |
|:---|:---:|:---:|:---|
| PC 浏览器（麦克风） | `mic_pc` | — | 现有功能，直连 center |
| PC Sensor（截屏 OCR） | — | `ocr_screen` | 截屏+本地OCR→文本→POST |
| PC Sensor（主动 OCR） | — | `ocr_proactive` | 监控微信/飞书→文本→POST |
| 录音豆 / 耳机 | `mic_hardware` | — | 音频流→WS→center ASR |
| 移动端（未来） | `mic_mobile` | `mobile_sensor` | 手机麦克风+传感器 |
| AI 眼镜（未来） | `mic_glasses` | `glasses_ocr` | 摄像头OCR+麦克风 |

## 五、使用方式

只有两种部署模式：

### 5.1 单机模式（不变）

```bash
python -m lifetrace.server
```

启动脚本沿用 `scripts/start-all.bat`，不做任何修改。

### 5.2 分布式模式（Center + Sensor，通过 cpolar）

两台机器，Center 通过 cpolar 暴露到公网，Sensor 从任意网络连接。

> **前置条件**：Center 机器已安装 cpolar 并完成认证（`cpolar authtoken <token>`）。
> 建议使用 cpolar 付费版获得固定子域名，免费版每次重启域名会变。

**步骤一：Center 节点**运行 `scripts/start-center.bat`

修改脚本顶部配置区：
```
set "CPOLAR_BACKEND_DOMAIN=my-lt-api"      → https://my-lt-api.cpolar.cn
set "CPOLAR_FRONTEND_DOMAIN=my-lt-ui"      → https://my-lt-ui.cpolar.cn
```

脚本自动完成：
1. 启动后端（`--role center`，绑定 `0.0.0.0:8001`）
2. 构建前端（`NEXT_PUBLIC_API_URL=https://my-lt-api.cpolar.cn`）并启动
3. 启动 cpolar 后端隧道（`cpolar http 8001 -subdomain=my-lt-api`）
4. 启动 cpolar 前端隧道（`cpolar http 3001 -subdomain=my-lt-ui`）

> cpolar 直接提供 HTTPS + WSS，不需要反向代理，不需要证书。
>
> **注意**：`NEXT_PUBLIC_API_URL` 在 Next.js 构建时固化，所以需要固定子域名。
> 如果使用免费版随机域名，需将脚本中的 `pnpm build:frontend:web && pnpm start`
> 改为 `pnpm dev`（dev 模式支持运行时读取环境变量）。

**步骤二：Sensor 节点**运行 `scripts/start-sensor.bat`

修改脚本顶部配置区：
```
set "CENTER_URL=https://my-lt-api.cpolar.cn"
set "CENTER_FRONTEND_URL=https://my-lt-ui.cpolar.cn"
```

脚本自动完成：
1. 检查 Center 连通性
2. 启动感知守护进程（截屏 + OCR + 主动 OCR → POST 到 Center）
3. 打开浏览器访问 Center 前端

```
数据流：
PC (任意网络)                          Center + cpolar
  sensor ──── HTTPS POST ────▶ my-lt-api.cpolar.cn/api/perception/ingest
  浏览器 ──── HTTPS ──────────▶ my-lt-ui.cpolar.cn (前端 UI)
  浏览器 ──── HTTPS ──────────▶ my-lt-api.cpolar.cn/api/* (数据 API)
  浏览器 ════ WSS ════════════▶ my-lt-api.cpolar.cn/ws/* (音频 + 感知流)
```

### 5.3 脚本一览

| 脚本 | 用途 | 在哪运行 |
|:---|:---|:---|
| `scripts/start-all.bat` | 单机一键启动（不变） | 单机模式 |
| `scripts/stop-all.bat` | 单机一键停止（不变） | 单机模式 |
| `scripts/start-center.bat` | 启动 Center（后端 + 前端 + cpolar） | Center 节点 |
| `scripts/stop-center.bat` | 停止 Center 全部服务 | Center 节点 |
| `scripts/start-sensor.bat` | 启动 Sensor + 打开浏览器 | PC 感知节点 |
| `scripts/stop-sensor.bat` | 停止 Sensor | PC 感知节点 |

### 6.2 `lifetrace/routers/perception_ingest.py`：新增文本感知接收端点

**新建文件**，Center 接收远程 Sensor 推送的 `PerceptionEvent`，注入本地 `PerceptionStream`。

```python
from fastapi import APIRouter
from pydantic import BaseModel

from lifetrace.perception.manager import get_perception_stream
from lifetrace.perception.models import PerceptionEvent

router = APIRouter(prefix="/api/perception", tags=["perception-ingest"])


@router.post("/ingest")
async def ingest_event(event: PerceptionEvent):
    stream = get_perception_stream()
    await stream.publish(event)
    return {"ok": True, "event_id": event.event_id, "sequence_id": event.sequence_id}


class BatchIngestRequest(BaseModel):
    node_id: str = ""
    events: list[PerceptionEvent]


@router.post("/ingest/batch")
async def ingest_batch(req: BatchIngestRequest):
    stream = get_perception_stream()
    for event in req.events:
        if req.node_id:
            event.metadata.setdefault("node_id", req.node_id)
        await stream.publish(event)
    return {"ok": True, "count": len(req.events)}
```

**代码量**：~40 行

### 6.5 `lifetrace/routers/audio_ws.py`：音频端点接受 source/node_id 参数

**改动内容**：

在 `websocket_transcribe` 函数中，从 WebSocket 的 query params 中解析 `source` 和 `node_id`，
传递给后续的 `PerceptionEvent` 生成逻辑。

**具体改动位置**：在 `audio_ws_handler.py` 的 `_handle_transcribe_ws` 或 `audio_ws.py` 的
`websocket_transcribe` 中：

```python
# 在 websocket accept 之后，解析 query params
source = websocket.query_params.get("source", "mic_pc")
node_id = websocket.query_params.get("node_id", "local")

# 后续生成 PerceptionEvent 时，将 source/node_id 写入 metadata
# 找到 try_publish_audio_transcription 的调用处，在 metadata 中追加：
metadata = {"source": source, "node_id": node_id, ...}
```

现有前端调用 `/ws/audio/transcribe` 时不带参数，默认值 `source=mic_pc, node_id=local`，
完全向后兼容。

**改动量**：~10 行

### 6.6 `lifetrace/sensor.py`：新增感知守护进程

**新建文件**，作为 Sensor 节点的入口（`python -m lifetrace.sensor`）。

不启动 FastAPI 服务器，只做：
1. 定时截屏 + 本地 OCR → 生成 `PerceptionEvent` → HTTP POST 到 Center
2. 主动 OCR（微信/飞书监控）→ 生成 `PerceptionEvent` → HTTP POST 到 Center
3. 定时心跳上报

**核心结构**：

```python
import argparse
import asyncio
import platform

import httpx

from lifetrace.perception.models import Modality, PerceptionEvent, SourceType


class SensorDaemon:
    """轻量感知守护进程，采集本地屏幕/OCR 数据并转发到 Center。"""

    def __init__(self, center_url: str, node_id: str):
        self.center_url = center_url.rstrip("/")
        self.node_id = node_id
        self.client = httpx.AsyncClient(timeout=10)
        self._event_buffer: list[PerceptionEvent] = []

    async def post_event(self, event: PerceptionEvent):
        """将单个 PerceptionEvent POST 到 Center。"""
        event.metadata["node_id"] = self.node_id
        url = f"{self.center_url}/api/perception/ingest"
        resp = await self.client.post(url, json=event.model_dump(mode="json"))
        resp.raise_for_status()

    async def post_batch(self, events: list[PerceptionEvent]):
        """批量 POST 到 Center。"""
        for e in events:
            e.metadata["node_id"] = self.node_id
        url = f"{self.center_url}/api/perception/ingest/batch"
        payload = {
            "node_id": self.node_id,

    async def run(self, screenshot_interval: float, proactive_ocr_interval: float):
        """主事件循环：并发运行截屏和主动 OCR 的定时任务。"""
        async def screenshot_loop():
            while True:
                try:
                    await self.run_screenshot_ocr_cycle()
                except Exception as e:
                    print(f"截屏 OCR 失败: {e}")
                await asyncio.sleep(screenshot_interval)

            while True:
                try:
                    await self.run_proactive_ocr_cycle()
                except Exception as e:
                    print(f"主动 OCR 失败: {e}")
                await asyncio.sleep(proactive_ocr_interval)

        async def heartbeat_loop():
            while True:
                await self.heartbeat()
                await asyncio.sleep(30)

        await asyncio.gather(
            screenshot_loop(),
            proactive_ocr_loop(),
            heartbeat_loop(),
        )
```

## 九、开放问题

1. **Sensor 断线处理**：Sensor 无法连接 Center 时，是否需要本地缓存感知事件？
   → MVP 阶段：丢弃，日志告警。后续可加本地 SQLite 缓存 + 重连后批量补发。

2. **多 Sensor 去重**：多台 PC Sensor 的事件是否可能重复？
   → 每个 Sensor 有唯一 `node_id`，事件携带 `event_id` (UUID)，Center 可按需去重。

3. **Sensor 配置同步**：Sensor 的 OCR 参数等是否需要从 Center 远程管理？
   → ✅ 已实现。Sensor 每 15s 轮询 `GET /api/sensor/config`，用户可在前端面板控制开关和间隔。

4. **截图文件存储**：Sensor 截取的截图是否需要上传到 Center？
   → MVP 阶段：只上传 OCR 文本（`PerceptionEvent.content_text`），截图留在 Sensor 本地。

5. **多节点感知流合并**：多台设备同时推送事件时，`sequence_id` 如何保证全局有序？
   → Center 侧的 `PerceptionStream.publish()` 已使用原子计数器分配 `sequence_id`，多来源注入自然有序。
